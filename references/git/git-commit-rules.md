# Git 커밋 규칙

## 커밋 메시지 작성 규칙

### 작업 컨텍스트가 1개인 경우

단일 작업 내용을 명확하게 작성합니다.

**예시:**
- `CLAUDE.md 한글로 변경`
- `로그인 기능 구현`
- `사용자 프로필 페이지 스타일 수정`

### 작업 컨텍스트가 2개인 경우

'및' 키워드를 사용하여 두 작업을 연결합니다.

**예시:**
- `CLAUDE.md 한글로 변경 및 camelCase 규칙 추가`
- `로그인 기능 구현 및 테스트 코드 작성`
- `API 엔드포인트 추가 및 문서 업데이트`

### 작업 컨텍스트가 3개 이상인 경우

커밋을 작업 컨텍스트별로 분리하여 여러 개의 커밋으로 쪼갭니다.
각 커밋은 하나의 논리적인 변경 단위를 포함해야 합니다.

**나쁜 예 (하나의 커밋에 3개 이상):**
```
A 기능 구현 및 B 기능 구현 및 C 기능 구현
```

**좋은 예 (각각 별도 커밋):**
```
커밋 1: A 기능 구현
커밋 2: B 기능 구현
커밋 3: C 기능 구현
```

## 커밋 단위 원칙

### 1. 단일 책임 (Single Responsibility)
- 하나의 커밋은 하나의 명확한 목적을 가져야 합니다
- 서로 관련 없는 변경사항은 별도의 커밋으로 분리합니다

### 2. 원자성 (Atomicity)
- 커밋은 그 자체로 완전하고 의미 있어야 합니다
- 커밋 후 프로젝트가 정상 작동해야 합니다
- 불완전한 작업은 커밋하지 않습니다

### 3. 가독성 (Readability)
- 커밋 히스토리를 읽었을 때 프로젝트 진행 과정이 명확해야 합니다
- 커밋 메시지만으로도 무엇이 변경되었는지 이해할 수 있어야 합니다
- 일관된 형식과 명명 규칙을 따릅니다

## 커밋 메시지 형식

### 기본 구조
```
[작업 내용을 명확히 설명하는 제목]

[선택사항: 상세 설명]
- 변경 이유
- 주요 변경 사항
- 영향 범위
```

### 제목 작성 가이드
- 한국어로 작성
- 명령형으로 작성 (예: "추가", "수정", "제거")
- 50자 이내로 간결하게
- 마침표 없이 작성

### 본문 작성 가이드 (선택사항)
- 제목만으로 설명이 부족한 경우 작성
- 무엇을(What) 변경했는지보다 왜(Why) 변경했는지 설명
- 72자마다 줄바꿈

## 예시

### 좋은 커밋 메시지

```
사용자 인증 로직 개선

- JWT 토큰 만료 시간을 24시간으로 연장
- 리프레시 토큰 자동 갱신 기능 추가
- 보안을 위해 토큰 블랙리스트 구현
```

```
404 에러 페이지 디자인 개선
```

### 나쁜 커밋 메시지

```
수정  # 무엇을 수정했는지 불명확
```

```
로그인 기능 추가 및 회원가입 페이지 수정 및 프로필 업데이트 기능 구현  # 3개 이상의 컨텍스트
```

```
Fixed bug  # 한국어 규칙 위반
```

## 커밋 전 체크리스트

- [ ] 커밋 메시지가 한국어로 작성되었는가?
- [ ] 작업 컨텍스트가 1-2개로 제한되어 있는가?
- [ ] 변경사항이 논리적으로 하나의 단위를 구성하는가?
- [ ] 커밋 후 프로젝트가 정상 작동하는가?
- [ ] 불필요한 파일이나 민감한 정보가 포함되지 않았는가?
